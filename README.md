# yadro-test-task-1

Тестовое задание для отбора на летнюю стажировку в Импульс YADRO.

## Сборка и запуск

Установка необходимых инструментов:
```bash
sudo apt install cmake gcc libgtest-dev
```

Проект собирается в отдельной папке build с помощью CMake
```bash
cmake -B build

cd build

make
```

Запуск программы:
```bash
./Task1 <file_name>
```

Запуск тестов:
```bash
make test
```

## Структура проекта

Решение проекта разделено по стандартным папкам:
- `include/` - заголовочные файлы, в которых описаны классы, используемые в программе. 
- `src/` - исходный код программы. Реализация каждого класса для удобства и удобочитаемости кода вынесена в отдельный файл.
- `tests/` - код для тестирования функционала программы

## Описание решения

Решение программы условно разделено на две части:
1. Внутренняя логика управления игрой. Она имплементирована в классы Resources,  Person, Room, Dungeon
2. Алгоритм решения поставленной задачи. Он реализован внутри классов, наследуемых от BaseBot.

При написании кода использовался паттерн Facade, чтобы сокрыть внутреннюю логику работы программы за вызовами необходимых методов.

### 1. Внутренняя логика

Сеть комнат в подземелье представлена в программе в виде неориентированного графа. Вершины графа представлены классом Room (комната). Она хранит в себе ресурсы и информацию о действиях персонажа в ней (сбор и посещение).

Сами объекты класса Room хранятся в отдельном списке в классе Dungeon
```bash
std::vector<Room> rooms_list_;
```

Для того, чтобы оптимизировать использование памяти, используется список смежностей, в котором элементы - номера комнат (uint8_t). Он представлен в классе Dungeon:
```cpp
std::vector<std::pair<uint8_t, std::vector<uint8_t>>> dungeon_;
```

В результате доступ к конкретной комнате можно получить путем `room_list_[room_number]`

Для хранения информации о персонаже используется класс Person. C помощью методов `enter()` и `collect()` происходит его управление.

Наконец, информация о ресурсах хранится в одном классе Resources. Для удобства доступа был перегружен оператор "[]"

Классы Resources, Room и Dungeon реализуют паттерн Strategy, что позволило управлять всем игровым миром через один класс Dungeon

### 2. Игровой алгоритм

Непосредственно решение поставленной задачи находится внутри класса AliseBot. Внутри него имплементирована логика, которую придумала Алиса, чтобы управлять игровым персонажем. 

Выбор решения в виде наследования от класса BaseBot обусловлен задачей, описанной в условии:
> Код программы должен быть написан так, чтобы в него было легко встроить и испытать любой другой алгоритм для бота, работающего в описанных условиях

Для этого был введен базовый класс бота, который бы задавал необходимые требования для последующих реализаций. Теперь, если будет необходимость изменить алгоритм работы, не нужно переписывать всю программу целиком. Достаточно лишь изменить внутренний код класса AliseBot, или в худшем случае написать новый класс NewBot.

Для реализации стратегии управления персонажем, предложенного Алисой, в классе AliseBot потребовалось реализовать два алгоритма:
1. Нахождение наименьшего пути между двумя комнатами, который проходил бы по уже пройденным комнатам (getShortestPath). Реализован через BFS
2. Нахождение наиболее ценного ресурса в комнате (getMostExpensiveResource). Реализован через простое сравнение.
